# System Design

Основано на презентации [Как подготовиться и пройти System Design Interview. Александр Поломодов](https://youtu.be/jUbOm0B-eKQ)

## Литература
- Джой Битти, Карл Вигерс - Software requirements. Third edition
- Software architecture for Busy developers
- Грегор Хоп, Бобби Вульф - Шаблоны интеграции корпоративных приложений
  Способы интеграции корпоративных приложений: Files, DBs, API, Messaging
- Эндрю Таненбаум - Компьютерные сети
- Мартин Клеппман - Высоконагруженные приложения. Программирование, масштабирование, поддержка (кабанчик)
- Форд, Ричардс, Садаладж, Дегани - Software Architecture: The Hard parts
- K8s Patterns
- Building Secure and Reliable Systems (книжка от Гугла)
- Алекс Петров - Database Internals
- Distributed Systems (немного устаревшие системы)

## Что стоит уметь
- Описать основной поток работы (happy path)
- Очертить exceptional flows
- Научиться определять read/write paths для системы
- Разделять рабочие нагрузки на stateful/stateless
- Проектировать модель данных
- Выбирать классы для stateful компонентов
  - RDBMS
  - NoSQL по основным типам: K/V, Document-oriented, Column-oriented, Graph
- Выбирать конкретных представителей систем из перечисленных в концептуальной схеме
- Думать в концепции failure domains
- Проектировать систему с учетом day-2 operations: эксплуатация и развитие (logs, monitoring, migrations etc)
- Масштабировать stateless компоненты
  - С использованием оркестраторов и HPA, VPA
- Масштабировать stateful компоненты
  - Масштабировать по чтению - Replication  
    Здесь могут быть ослаблены требования по consistency
  - Масштабировать по записи - Partitioning и Sharding  
    Здесь нужен определенный паттерн доступа к данным, чтобы схема работала

## Что стоит изучить
- Container Diagram и System Context Diagram из C4 model - c4model.com
- Sequence и Activity Diagrams из UML
- IDEF0 и DFD диаграммы
- BPMN диаграммы
- Как балансировать read/write path для системы. Хорошо описано в книге Клеппмана
- Как делить систему на части - Domain Driven Design и конкретно виды subdomains и Bounded Context
- Как моделировать данные ER Diagrams, Class Diagrams
- Принципы для stateless приложений из The Twelve-Factor App
- Какие БД бывают и границы их применимости
- Как работают оркестраторы рабочих нагрузок, навроде k8s
- Потрогать руками основных представителей систем из перечисленных в концептуальной схеме
- Понять гарантии, которые дают эти решения и возможный диапазон рабочих нагрузок, которые они потянут
- Изучить SRE подходы и потрогать руками инструменты для logs, monitoring, migrations etc
- Посчитать и погонять нагрузочные тесты с любимыми вашими инструментами

## Советы к интервью
- Помнить о времени
- Не начинать проектировать сразу
- Делиться размышлениями при проектировании с интервьюером
- Проявлять самостоятельность в проектировании - вести интервью
- Реагировать на наводящие вопросы и уточнения
- Говорить уверенно о том, что знаешь, а не то слышал
- Подготовить вопросы к интервьюеру на случай, если для них останется время

## Шаги на интервью по Системному дизайну
- Понять задачу и определить масштаб решения
- Предложить общее решение и получить согласие
- Подробное проектирование
- Подведение итогов
